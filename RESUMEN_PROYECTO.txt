¡Hola! Aquí te cuento un poco cómo abordamos y trabajamos este proyecto de TodoCamisetas, paso a paso.

### 🛠️ El Stack Tecnológico
Para este proyecto decidimos irnos por un enfoque clásico y robusto, sin depender de frameworks mágicos. Usamos:
- **PHP 8 Puro:** Para toda la lógica del backend. Nada de Laravel o Symfony; aquí construimos nuestro propio enrutador y controladores desde cero.
- **MySQL:** Como nuestra base de datos relacional.
- **Nginx:** Como servidor web para manejar las peticiones HTTP.
- **Docker & Docker Compose:** Contenerizamos todo para que sea súper fácil de levantar en cualquier máquina sin ensuciar el sistema operativo local. Literalmente con un `docker compose up -d` ya tienes todo andando.

### 🌊 El Flujo de Trabajo (Paso a Paso)

1. **La Base y la Arquitectura (Fases 1 y 2):** 
   Primero montamos la infraestructura con Docker. Creamos los contenedores para Nginx, PHP y MySQL. Luego, definimos una estructura de carpetas limpia: `src/controllers`, `src/database/migrations`, `src/models`, `src/routes`, etc. Así sabemos exactamente dónde va cada cosa.

2. **El Corazón: El Enrutador:**
   Creamos un `router.php` personalizado que usa expresiones regulares para manejar las rutas y los métodos HTTP (GET, POST, PUT, DELETE), apuntando directamente a los métodos de nuestros controladores.

3. **Las Migraciones (Fase 0 y Correcciones):**
   Armamos scripts SQL (`.sql`) para crear las tablas (`camisetas`, `clientes`, `tallas`, etc.). Nos dimos cuenta leyendo el PDF del examen que faltaban campos clave (como el título, país, tipo y color de la camiseta, o la dirección y categoría exacta del cliente). Así que, lo primero que hicimos fue ajustar estas migraciones para que calzaran perfecto con lo que pedía el cliente.

4. **Modelos y Controladores (Fases 5 y 7):**
   - **El Problema:** Al inicio, las consultas SQL (los INSERT, SELECT) estaban incrustadas directamente en los Controladores. Esto no escalaba y violaba el principio de responsabilidad única.
   - **La Solución:** Refactorizamos el código implementando **Modelos** (`Camiseta.php`, `Cliente.php`, `Orden.php`, `Talla.php`) usando PDO para manejar exclusivamente la conexión a datos (métodos estáticos como `all()`, `find()`, `create()`, etc.). Luego, limpiamos los **Controladores** (`CamisetaController`, etc.) para que se dediquen solo a recibir el request, validar estrictamente los campos (ej. que la camiseta sea 'Local' o 'Visita', y que el cliente sea 'Regular' o 'Preferencial') y llamar a los modelos correspondientes.

5. **El Reto Principal: La Logística de Precios (Fase 6):**
   - ¡La parte más crítica del examen está lista! Creamos el método `getPrecioFinal` y registramos la nueva ruta `GET /api/camisetas/{id}/precio-final?cliente_id=X` en el archivo `router.php`. 
   - Esta ruta aplica la regla de negocio consultando primero al modelo del Cliente y luego al de la Camiseta correspondientes. Si el cliente consultando era "Preferencial" (como 90minutos) y la camiseta tenía una oferta, le dábamos ese precio. Si era "Regular" o no había oferta, pagaba el precio normal. Todo resuelto del lado del backend sin tocar el front.

### 🧪 Miniguía: ¿Cómo hacer las pruebas?

Una vez que todo esté corriendo (puedes verificarlo con `docker compose ps`), puedes usar herramientas como **Postman**, **Insomnia**, o tu terminal con **cURL** para probar la API. 

Aquí te dejo unos ejemplos rápidos:

**1. Revisar que la API está viva (Health Check)**
```bash
curl http://localhost:8080/api/health
```

**2. Crear una Camiseta nueva (POST)**
```bash
curl -X POST http://localhost:8080/api/camisetas \
  -H "Content-Type: application/json" \
  -d '{
    "sku": "SCL2025L",
    "titulo": "Camiseta Local 2025 - Selección Chilena",
    "club": "Selección Chilena",
    "pais": "Chile",
    "tipo": "Local",
    "color": "Blanco y Negro",
    "precio": 45000,
    "precio_oferta": 35000,
    "detalles": "Edición aniversario 2025"
  }'
```

**3. Crear un Cliente (POST)**
```bash
curl -X POST http://localhost:8080/api/clientes \
  -H "Content-Type: application/json" \
  -d '{
    "rut": "12.345.678-9",
    "nombre_comercial": "90minutos",
    "direccion": "Providencia, Santiago",
    "categoria": "Preferencial",
    "contacto_nombre": "Juan Pérez",
    "contacto_email": "juan@90minutos.cl",
    "descuento_porcentaje": 5
  }'
```

**4. Probar la Lógica del Precio Final (GET)**
Para ver la magia en acción, asumiendo que la camiseta tiene ID 1 y el cliente Preferencial tiene ID 1:
```bash
curl "http://localhost:8080/api/camisetas/1/precio-final?cliente_id=1"
```
*(Debería devolverte el JSON indicando que se aplicó el precio de oferta de 35000).*

¡Y listo! Con esto tienes un pantallazo general de lo que se construyó y cómo probarlo.
